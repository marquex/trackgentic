# User Authentication

Trackgentic is designed to be used by multiple actors — AI agents, humans, or automated systems. To know who is performing each action, we introduce a lightweight authentication system based on simple tokens. This is not cryptographically secure — it's a practical mechanism to attribute actions to users and prevent accidental impersonation.

## How it works

1. A user registers with a name using `trackgentic users register`.
2. The system generates a short random token and stores it alongside the user name.
3. Depending on the configured auth mode, the user may need to provide the token when executing commands.
4. The system resolves the token to the user name and records it in the event.

## User registry file

Registered users are stored in `.trackgentic/users.json`. This file is created automatically during `init` as an empty registry.

```json
{
  "users": [
    { "name": "alice", "token": "tk_k7x2m9p4", "registeredAt": "2024-06-01T12:00:00Z" },
    { "name": "bob", "token": "tk_n3w8f1q6", "registeredAt": "2024-06-02T10:00:00Z" }
  ]
}
```

### Token format

Tokens are prefixed with `tk_` followed by 8 random alphanumeric characters (e.g., `tk_k7x2m9p4`). This is short enough to be practical and unique enough to avoid collisions in normal usage. The prefix makes tokens easy to identify as trackgentic tokens when seen in configuration or logs.

## Auth mode

The behavior of the system when a command is executed without a token is configurable. The auth mode is set in `.trackgentic/config.json` under the `auth` key.

```json
{
  "auth": {
    "mode": "read-only",
    "defaultUser": "anonymous"
  }
}
```

There are three modes:

### `open`

No token is required for any command. If a mutating command is executed without a token, the action is attributed to the `defaultUser` configured in `config.json` (defaults to `"anonymous"`). If a token is provided, it is validated and the resolved user is used instead.

This mode is useful for single-user setups, local development, or when attribution is nice-to-have but not essential.

If `mode` is set to `open` but `defaultUser` is missing or empty, any command will fail with a `DEFAULT_USER_MISSING` error.

### `read-only` (default)

Read-only commands (`list`, `view`, `history`, `comments list`, `blockages list`) do not require a token. Mutating commands (`create`, `update`, `comments add/update/delete`, `blockages add/resolve/delete`) require a valid token. If a mutating command is executed without a token, it fails with a `TOKEN_REQUIRED` error.

This is the default mode, balancing convenience for querying with accountability for changes.

### `strict`

All commands require a valid token — both reads and writes. Any command executed without a token fails with a `TOKEN_REQUIRED` error. This is useful when you want full traceability of who is even looking at issues.

### Summary

| Mode | Reads without token | Writes without token |
|------|--------------------|--------------------|
| `open` | Allowed | Allowed (attributed to `defaultUser`) |
| `read-only` | Allowed | Rejected |
| `strict` | Rejected | Rejected |

## Passing the token

Users identify themselves by setting the `TRACKGENTIC_USER_TOKEN` environment variable to their token before running commands. This keeps token management out of command-line arguments (where tokens could end up in shell history or process listings) and makes it easy for agents and scripts to authenticate once per session.

```bash
export TRACKGENTIC_USER_TOKEN=tk_k7x2m9p4
trackgentic create "Fix login bug" --status todo
```

## Author field in events

Every event generated by a mutating command includes an `author` field with the resolved user name:

```json
{ "timestamp": "2024-06-01T12:00:00Z", "type": "update", "author": "alice", "content": { "status": "in-progress" } }
```

This applies to all event types: `creation`, `update`, `comment`, `comment-update`, `comment-delete`, `blockage-added`, `blockage-resolved`, `blockage-deleted`.

When running in `open` mode without a token, the `author` field is set to the configured `defaultUser`.

## CLI commands

### `trackgentic users register [name]`

Register a new user with the given name. The name must be unique among registered users.

**Returns:**
- `{ "result": "OK", "name": "[name]", "token": "[token]" }` if registration was successful.
- `{ "result": "USER_ALREADY_EXISTS", "message": "A user with name '[name]' is already registered" }` if the name is taken.

**Example:**
```bash
trackgentic users register alice
# returns { "result": "OK", "name": "alice", "token": "tk_k7x2m9p4" }
```

**Notes:**
* User names are case-insensitive and stored lowercase.
* This command does not require a token (it's how users obtain their token).
* The name `anonymous` is reserved for the default unauthenticated user and cannot be registered.

### `trackgentic users list`

List all registered users.

**Returns:**
An array of users, each with:
* `name`: the user name
* `registeredAt`: when the user was registered

**Example:**
```bash
trackgentic users list
# returns [
#   { "name": "alice", "registeredAt": "2024-06-01T12:00:00Z" },
#   { "name": "bob", "registeredAt": "2024-06-02T10:00:00Z" }
# ]
```

**Notes:**
* Tokens are **not** included in the list output.
* This command does not require a token (subject to auth mode).

### `trackgentic users revoke [name]`

Revoke a user's token, effectively deregistering them. The user will no longer be able to authenticate.

**Returns:**
- `{ "result": "OK" }` if revocation was successful.
- `{ "result": "USER_NOT_FOUND", "message": "No user found with name '[name]'" }` if the user doesn't exist.

**Example:**
```bash
export TRACKGENTIC_USER_TOKEN=tk_k7x2m9p4
trackgentic users revoke bob
# returns { "result": "OK" }
```

**Notes:**
* Requires a valid token via the `TRACKGENTIC_USER_TOKEN` environment variable (the caller must be a registered user).

### `trackgentic users regenerate [name]`

Generate a new token for an existing user, invalidating the old one. Useful if a token is accidentally exposed.

**Returns:**
- `{ "result": "OK", "name": "[name]", "token": "[new_token]" }` with the new token.
- `{ "result": "USER_NOT_FOUND", "message": "No user found with name '[name]'" }` if the user doesn't exist.
- `{ "result": "INVALID_TOKEN", "message": "The provided token does not match the user" }` if the token doesn't match.

**Example:**
```bash
export TRACKGENTIC_USER_TOKEN=tk_k7x2m9p4
trackgentic users regenerate alice
# returns { "result": "OK", "name": "alice", "token": "tk_r5t1y8u2" }
```

**Notes:**
* The `TRACKGENTIC_USER_TOKEN` must contain the user's current token (only the user itself can regenerate its token).

## Configuration

Auth configuration lives in `.trackgentic/config.json`, which is created during `init` with the default settings:

```json
{
  "auth": {
    "mode": "read-only",
    "defaultUser": "anonymous"
  }
}
```

The `mode` can be changed manually by editing the file. The `defaultUser` is only used when `mode` is `open` and a command is executed without a token.

## Errors

### Auth errors

* `TOKEN_REQUIRED`: A command was called without a token and the auth mode requires one.
* `INVALID_TOKEN`: The provided token does not match any registered user.
* `DEFAULT_USER_MISSING`: The authentication system is open for writing but there is no default user defined in config.json to attribute unauthenticated changes.
* `USER_ALREADY_EXISTS`: Attempted to register a user with a name that is already taken.
* `USER_NOT_FOUND`: Attempted to revoke or regenerate a token for a user that doesn't exist.

## Folder structure

```
.trackgentic/
  config.json
  index.json
  dependencies.json
  users.json
  issues/
    [issue_id].json
```

## Impact on existing commands

Whether a token is required depends on the auth mode. The user name resolved from the `TRACKGENTIC_USER_TOKEN` environment variable is recorded as `author` in every event. This means:

* **`create`**: The `creation` and initial `update` events include the `author` field.
* **`update`**: The `update` event includes the `author` field.
* **`comments add/update/delete`**: The comment events include the `author` field. The author of a comment is always the authenticated user.
* **`blockages add/resolve/delete`**: The blockage events include the `author` field.

For commands that don't require a token (depending on auth mode), the `author` field is omitted from events if no token was set — except in `open` mode, where it defaults to `defaultUser`.
